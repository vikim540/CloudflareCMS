## 一、開發階段劃分

項目分三個階段推進,每個階段有明確的目標、任務清單和驗收標準。階段之間允許部分重疊,但前一階段的驗收標準必須全部達成後才能進入下一階段的核心任務。

### 1.1 第一階段:MVP 最小可用版本(預計 2-3 週)

**目標:** 完成核心 API + 數據庫 + 基礎鑑權,可實現後台發布文章、前台通過 API 拉取數據。

本階段聚焦最小可用閉環:管理員能登錄、能創建欄目和文章、前台 API 能拉取數據。不追求功能齊全,只驗證技術鏈路完整貫通。

#### 任務清單

| 序號 | 任務 | 涉及文件/接口 | 說明 |
|------|------|--------------|------|
| 1 | 項目初始化 | `Cargo.toml`、`wrangler.toml`、`migrations/001_init.sql` | 創建 Cargo 項目,配置 `wrangler.toml`(D1/KV/R2 binding),編寫 D1 建表 SQL(對齊 PbootCMS 表結構) |
| 2 | Worker 入口和路由框架搭建 | `src/lib.rs`、`src/router.rs` | 使用 `worker` crate 的 Router,註冊 `/api/v1/` 路由分組,劃分公開接口、管理接口、SSG 接口三層 |
| 3 | JWT 自行實現(sha2 + hmac,HS256) | `src/util/jwt.rs` | 基於 `sha2` + `hmac` crate 實現 HS256 簽發與驗證,代碼量不超過 100 行,使用常量時間比較防止時序攻擊。參考 RFC 7519 規範 |
| 4 | 雙 MD5 密碼實現 | `src/util/password.rs` | 實現 `md5(md5(password))` 以兼容 PbootCMS 原版 `ay_user` 數據,使用常量時間比較 |
| 5 | 統一響應格式和錯誤處理 | `src/util/response.rs`、`src/middleware/error.rs` | 實現 `{code, msg, data, meta}` 統一響應格式,定義錯誤碼(0 成功,1001-1005 通用,2001-2003 認證),業務錯誤統一返回 HTTP 200 |
| 6 | 管理員登錄接口 | `POST /api/v1/auth/login` | 驗證用戶名密碼(雙 MD5 比對),簽發 JWT Token,返回 token + 用戶信息。含基於 KV 的 IP 粒度登錄限頻(5 次失敗鎖定 15 分鐘) |
| 7 | JWT 鑑權中間件 | `src/middleware/auth.rs` | 解析 `Authorization: Bearer <token>`,驗證簽名和過期時間,從 KV 檢查 Token 黑名單,提取用戶信息注入請求上下文。超級管理員(uid=1)跳過權限檢查 |
| 8 | D1 數據庫操作封裝 | `src/service/db.rs` | 封裝 D1 binding API 的 `prepare/bind/all/first/run` 方法,提供類型安全的查詢接口,手動映射結果集(零反射) |
| 9 | KV 緩存封裝 | `src/service/cache.rs` | 封裝 KV 的 `get/put/delete` 方法,支持 TTL 設置,提供緩存命中/未命中統一處理邏輯 |
| 10 | 系統配置讀取 | `src/service/config.rs`、`GET /api/v1/admin/config` | 從 KV 緩存讀取配置(`config:all`),未命中時回退 D1 查詢 `ay_config` 表並寫入 KV。配置修改時刪除緩存觸發回源 |
| 11 | 欄目管理 CRUD | `/api/v1/admin/sorts`、`src/handler/sort.rs`、`src/service/sort.rs` | 實現欄目列表(樹形)、詳情、新增、修改、刪除。刪除時校驗子欄目。使用遞歸 CTE(`WITH RECURSIVE`)一次查詢子孫欄目,替代 N 次遞歸 |
| 12 | 內容管理 CRUD | `/api/v1/admin/contents`、`src/handler/content.rs`、`src/service/content.rs` | 實現內容列表(分頁)、詳情(含擴展字段)、新增、修改、刪除(移入回收站)。擴展字段讀寫 `ay_content_ext` + `ay_extfield`,批量查詢消除 N+1 |
| 13 | 前台公開接口 | `/api/v1/sorts`、`/api/v1/contents`、`/api/v1/contents/:id`、`/api/v1/nav` | 實現欄目列表、欄目詳情(scode/filename/urlname 三選一)、內容列表(分頁+篩選)、內容詳情(含上下篇)、導航樹(遞歸構建 children)。無需鑑權 |
| 14 | 基礎部署驗證 | `wrangler deploy` | 部署到 Cloudflare Workers,驗證 D1/KV binding 正常,線上接口可用 |

#### 驗收標準

- [ ] 管理員可通過 `POST /api/v1/auth/login` 登錄並獲取 JWT Token
- [ ] 可通過管理接口創建欄目(`ay_content_sort`)和文章(`ay_content` + `ay_content_ext`)
- [ ] 前台公開 API(`/api/v1/sorts`、`/api/v1/contents`、`/api/v1/contents/:id`)可正確拉取數據
- [ ] JWT 鑑權中間件正確攔截未授權請求(返回 code=1002)
- [ ] KV 緩存正確工作(配置讀取命中 KV,修改後緩存失效)
- [ ] 編譯後 WASM 體積 < 800KB
- [ ] `wrangler deploy` 部署成功,線上功能驗證通過

### 1.2 第二階段:功能完善(預計 3-4 週)

**目標:** 補齊所有業務模塊,完成管理後台 SPA。

本階段在 MVP 基礎上,逐一實現剩餘業務模塊的 CRUD 接口,並開發 React 管理後台 SPA,實現全功能可視化管理。同時完成靜態生成專用接口,供靜態生成層調用。

#### 任務清單

| 序號 | 任務 | 涉及文件/接口 | 說明 |
|------|------|--------------|------|
| 1 | 單頁管理模塊 | `/api/v1/admin/singles`、`src/handler/single.rs` | 單頁 CRUD,對應 `ay_single` 表。用於「關於我們」、「聯繫我們」等靜態頁面。前台公開接口 `/api/v1/singles`、`/api/v1/singles/:scode` |
| 2 | 附件管理(R2 上傳/列表/刪除) | `/api/v1/admin/upload`、`/api/v1/admin/attachments`、`src/handler/attachment.rs` | 文件上傳至 R2(`BUCKET` binding),記錄元信息。支持 `multipart/form-data` 上傳,按日期分目錄存儲。刪除時同時刪除 R2 對象。不做圖片處理(水印交由靜態生成層) |
| 3 | 友情連結 CRUD | `/api/v1/admin/links`、`src/handler/link.rs` | 友情連結 CRUD,對應 `ay_link` 表。前台公開接口 `/api/v1/links?gid=` |
| 4 | 幻燈片 CRUD | `/api/v1/admin/slides`、`src/handler/slide.rs` | 幻燈片 CRUD,對應 `ay_slide` 表。前台公開接口 `/api/v1/slides?gid=` |
| 5 | 標籤/內鏈 CRUD | `/api/v1/admin/tags`、`src/handler/tags.rs` | 標籤 CRUD,對應 `ay_tags` 表。用於 SEO 內鏈建設。前台公開接口 `/api/v1/tags` |
| 6 | 自定義標籤 CRUD | `/api/v1/admin/labels`、`src/handler/label.rs` | 自定義標籤 CRUD,對應 `ay_label` 表。用於在模板中插入動態內容。前台公開接口 `/api/v1/labels` |
| 7 | 留言表單(前台提交 + 後台管理) | `POST /api/v1/messages`、`/api/v1/admin/messages`、`src/handler/message.rs` | 前台提交留言(限頻,同一 IP 每分鐘 5 次),對應 `ay_message` 表。後台管理:列表(分頁+狀態篩選)、詳情、回覆、修改狀態、刪除、批量操作 |
| 8 | 自定義表單模塊 | `/api/v1/admin/forms`、`src/handler/form.rs` | 表單定義 CRUD(`ay_form`)+ 字段管理(`ay_form_field`)+ 表單數據查詢/刪除。支持動態擴展字段 |
| 9 | 301 重定向管理 | `/api/v1/admin/redirects`、`src/handler/redirect.rs` | 重定向規則 CRUD,對應 `ay_301_redirect` 表。Workers 原生支持 301 重定向,可結合路由層實現 |
| 10 | 系統日誌(輕量) | `/api/v1/admin/syslogs`、`src/handler/syslog.rs` | 系統日誌列表、詳情、清除。對應 `ay_syslog` 表。僅記錄關鍵操作(登錄、內容發布/刪除、配置修改),使用 `waitUntil` 異步寫入,不阻塞響應。減少 D1 寫入壓力 |
| 11 | 角色權限管理(簡化 RBAC) | `/api/v1/admin/roles`、`src/handler/role.rs` | 角色 CRUD + 權限設置,對應 `ay_role` + `ay_role_level`。基於 URL 級權限校驗,超級管理員(uid=1)跳過所有權限檢查 |
| 12 | 菜單管理 | `/api/v1/admin/menus`、`src/handler/menu.rs` | 菜單 CRUD(樹形)+ 菜單操作管理,對應 `ay_menu` + `ay_menu_action`。菜單數據供管理後台渲染側邊欄 |
| 13 | 站點/公司信息管理 | `/api/v1/admin/site`、`/api/v1/admin/company`、`src/handler/site.rs` | 站點信息(`ay_site`)和公司信息(`ay_company`)的讀取和修改。修改後同步更新 KV 緩存(`site:info`) |
| 14 | 靜態生成專用接口(SSG) | `/api/v1/ssg/*`、`src/handler/ssg.rs` | 實現全量導出(`export-all`)、增量更新(`incremental`)、文章分批拉取(`contents/batch`)、文章完整詳情(`contents/:id/full`)、全站配置(`config`)、sitemap 生成。API Key 驗證,與管理員 Token 體系隔離 |
| 15 | React 管理後台 SPA 開發 | `admin/` 目錄(獨立項目) | React 18 + TypeScript + Vite + shadcn/ui + Tailwind CSS。實現登錄頁、佈局框架、欄目管理、內容管理、單頁管理、附件管理、留言管理、系統配置等全部模塊的前端界面。使用 fetch 封裝 HTTP 客戶端,處理 JWT 鑑權和統一響應格式 |
| 16 | Cloudflare Pages 部署 | `wrangler pages deploy` | 管理後台 SPA 構建後部署到 Cloudflare Pages,綁定獨立域名(如 `admin.example.com`)。不佔用 Worker 資源 |

#### 驗收標準

- [ ] 所有業務模塊 CRUD 接口可用(單頁、附件、友情連結、幻燈片、標籤、自定義標籤、留言、自定義表單、301重定向、系統日誌、角色、菜單、站點/公司信息)
- [ ] 管理後台 SPA 可完整操作所有模塊(登錄、增刪改查、批量操作、文件上傳)
- [ ] SSG 接口可供靜態生成服務調用(全量導出、增量更新、分批拉取、完整詳情、全站配置、sitemap)
- [ ] 角色權限校驗生效(非超級管理員只能訪問授權的菜單和操作)
- [ ] 管理後台 SPA 成功部署到 Cloudflare Pages
- [ ] WASM 體積仍 < 800KB(新增模塊後複查)

### 1.3 第三階段:優化與完善(預計 2 週)

**目標:** 性能優化、體積壓縮、體驗提升。

本階段在功能齊全的基礎上,針對 Cloudflare Workers 環境的約束(WASM 體積、CPU 時間、D1 寫入限制)進行專項優化,並完善數據遷移、安全防護和文檔。

#### 任務清單

| 序號 | 任務 | 涉及文件/接口 | 說明 |
|------|------|--------------|------|
| 1 | WASM 體積優化 | `Cargo.toml`(`[profile.release]`) | 啟用 LTO(`lto = true`)、`opt-level = "z"`、`codegen-units = 1`、`panic = "abort"`、`strip = true`。編譯後用 `wasm-opt -Oz` 壓縮。逐一審查依賴,裁剪不必要的 transitive dependencies 和 feature flag。目標 < 800KB,力爭 < 500KB |
| 2 | KV 緩存策略完善 | `src/service/cache.rs` | 精準緩存失效:欄目增刪改時刪除 `sorts:tree` + `nav:tree`;文章修改時刪除 `content:{id}`;配置修改時刪除 `config:all`。避免過度失效導致緩存命中率下降。增加緩存命中日誌 |
| 3 | API 響應速度優化 | 各 Service 層 | 減少 D1 查詢次數:配置/欄目樹優先讀 KV;文章列表分頁後批量查詢擴展字段和欄目名稱(消除 N+1);統計總數時清除 ORDER BY;高頻查詢字段建立複合索引(索引覆蓋,避免回表) |
| 4 | 批量操作接口 | `POST /api/v1/admin/contents/batch` 等 | 批量移動欄目、批量修改狀態(發布/草稿)、批量置頂/推薦、批量刪除。使用 D1 batch 減少請求次數 |
| 5 | 數據導入導出(從 PbootCMS 遷移) | 遷移腳本(獨立工具) | 編寫遷移工具,從 PbootCMS MySQL/SQLite 導出數據,轉換並導入 D1。處理表結構差異、字符編碼(統一 UTF-8)、密碼格式(雙 MD5 直接遷移)、擴展字段映射 |
| 6 | Cloudflare Turnstile 集成 | 留言提交接口 | 集成 Cloudflare Turnstile 替代傳統驗證碼,防止垃圾留言。Workers 原生支持,無需 wasm 實現。前台表單嵌入 Turnstile widget,後台驗證 token |
| 7 | 錯誤處理完善 | `src/middleware/error.rs` | 統一捕獲所有 panic 和錯誤,返回規範的錯誤響應。完善錯誤碼覆蓋,增加上下文信息。D1 查詢錯誤、KV 操作錯誤、參數校驗錯誤分類處理 |
| 8 | 日誌和監控 | `src/middleware/log.rs` | 請求日誌記錄(方法、路徑、耗時、狀態碼),使用 `console_log!` 輸出到 Workers 日誌。關鍵操作寫入 `ay_syslog`。結合 Cloudflare Analytics 監控請求量和錯誤率 |
| 9 | 文檔完善 | `docs/` 目錄 | 完善 API 文檔、部署文檔、數據庫遷移文檔。補充接口請求/響應示例。編寫管理後台使用手冊 |

#### 驗收標準

- [ ] WASM 體積 < 800KB(力爭 < 500KB),`wasm-opt` 壓縮後部署正常
- [ ] KV 緩存命中率 > 90%(配置、欄目樹等熱點數據)
- [ ] API 平均響應時間 < 50ms(排除網絡傳輸)
- [ ] 批量操作接口可用,單次請求可處理 100 條以上記錄
- [ ] PbootCMS 數據遷移工具可用,遷移後數據完整、密碼可登錄
- [ ] Turnstile 集成生效,垃圾留言顯著減少
- [ ] 所有錯誤返回規範的統一響應格式,無裸 500 錯誤
- [ ] 日誌可查,監控數據可觀測
- [ ] 文檔齊全,新開發者可獨立參照文檔完成部署和二次開發

## 二、技術風險與應對

本項目基於 Cloudflare Workers + Rust + wasm32-unknown-unknown 目標,存在若干平台特定風險。以下逐項分析影響程度和應對措施,開發過程中需持續關注。

| 風險 | 影響 | 應對措施 |
|------|------|---------|
| WASM 體積超限 | Workers 免費版限制 1MB(壓縮後),超限無法部署 | 嚴格控制依賴,啟用 LTO 和 `wasm-opt -Oz`;不引入 `sqlx`/`jsonwebtoken`/`image`/`tokio` 等重依賴;自行實現 JWT HS256;每個 PR 檢查編譯體積;目標 < 800KB,保留安全餘量 |
| D1 寫入限制 | 免費版每天 100,000 行寫入,超限功能不可用 | 熱點數據緩存至 KV(配置、欄目樹),減少讀寫 D1;系統日誌僅記錄關鍵操作,使用 `waitUntil` 異步寫入;批量操作使用 D1 batch;文章訪問量累加可考慮 KV 計數器 + 定時回寫 |
| CPU 時間超限(10ms) | 免費版 CPU 時間限制 10ms,超限請求失敗 | 優化 D1 查詢(遞歸 CTE、批量查詢、索引覆蓋);減少計算密集操作(無圖片處理、無模板渲染);配置/欄目樹等熱點數據走 KV(KV 讀取不計 CPU);複雜查詢分頁控制數據量;必要時升級 Workers 付費版(50ms CPU) |
| JWT 自行實現的安全風險 | 實現不當可能導致認證漏洞(簽名繞過、時序攻擊) | 參考 RFC 7519 規範實現;簽名驗證使用常量時間比較(`subtle` crate 或手動實現);嚴格校驗 `exp` 過期時間和 `alg` 字段;Token 使用獨立 `jti` 標識支持黑名單;JWT_SECRET 通過 `wrangler secret` 設置,不硬編碼 |
| D1 不支持某些 SQLite 特性 | 遞歸 CTE、FTS5、某些函數可能行為不同,查詢失敗 | 使用 D1 兼容語法,提前在 `wrangler dev` 本地測試;遞歸 CTE 查子孫欄目先驗證可用性;全文搜索降級為 SQL `LIKE`,未來再評估 FTS5;關注 Cloudflare D1 更新日誌,及時適配新特性 |
| R2 大文件上傳超時 | Workers 請求體大小限制(免費版 100MB),大文件上傳可能超時 | 限制單文件大小(如 10MB 以內);大文件使用 R2 multipart upload;前端做上傳進度提示和重試;圖片類型可引導使用 Cloudflare Image Resizing |
| KV 最終一致性 | KV 寫入後全球傳播可能延遲數秒至數十秒,緩存刪除後短暫讀到舊值 | 對一致性要求高的場景(如 Token 黑名單)結合 D1 校驗;緩存刪除後設置短 TTL 兜底;關鍵配置修改後提示用戶稍候生效 |

## 三、依賴檢查清單

開發前需確認以下工具和環境已正確安裝配置。建議在開始第一階段任務 1(項目初始化)之前逐項驗證。

### Rust 工具鏈

| 工具 | 要求 | 驗證命令 | 說明 |

## 四、測試策略

採用分層測試策略,從單元測試到部署測試逐層驗證,確保每個階段交付物的質量。

### 4.1 單元測試

針對純邏輯函數編寫單元測試,不依賴 Workers 運行時,可在本地直接 `cargo test` 運行。

| 測試對象 | 測試內容 | 測試要點 |
|---------|---------|---------|
| JWT 簽發驗證(`src/util/jwt.rs`) | Token 簽發、驗證、過期判斷、篡改檢測 | 正確 payload 簽發後可驗證通過;過期 Token 驗證失敗;篡改 payload 後簽名不匹配;非法格式 Token 返回 None |
| 密碼比對(`src/util/password.rs`) | 雙 MD5 計算、密碼比對 | 已知密碼的 MD5 值正確;比對正確密碼返回 true;比對錯誤密碼返回 false;空密碼處理 |
| 分頁計算(`src/util/paginate.rs`) | 頁碼計算、偏移量計算、邊界處理 | page=1/pagesize=20 正確計算 offset=0;page=0 自動修正為 1;pagesize=0 或超限自動修正;total=0 時總頁數為 0 |
| 響應格式(`src/util/response.rs`) | 成功響應、錯誤響應、分頁響應構造 | 成功響應 code=0;錯誤響應 data=null;分頁響應 meta 字段正確 |

### 4.2 集成測試

使用 `wrangler dev` 啟動本地開發環境,測試 D1 查詢和 KV 操作的真實行為。

| 測試對象 | 測試方式 | 測試要點 |
|---------|---------|---------|
| D1 數據庫操作 | `wrangler dev --local` + 本地 D1 | 建表 SQL 執行成功;CRUD 操作正確;遞歸 CTE 查詢子孫欄目結果正確;分頁查詢 offset/limit 正確;擴展字段讀寫正確 |
| KV 緩存操作 | `wrangler dev --local` + 本地 KV 模擬 | 配置讀取命中 KV;KV 未命中回退 D1 並寫入 KV;緩存刪除後下次讀取回源;TTL 設置生效 |
| JWT 鑑權中間件 | `wrangler dev` + 模擬請求 | 攜帶有效 Token 請求通過;未攜帶 Token 返回 1002;攜帶過期 Token 返回 2002;攜帶黑名單 Token 返回 1002 |

### 4.3 API 測試

對所有接口進行端到端測試,驗證請求/響應符合 API 設計文檔規範。
