# 靜態生成層對接

> 本文件定義靜態生成層（SSG, Static Site Generation）與後端 Cloudflare Workers API 的完整對接方案。靜態生成服務部署於香港自有伺服器,通過 HTTP 拉取 Worker API 數據後生成全靜態 HTML,再同步至 CDN 分發。本文檔與 [03-API接口設計.md](./03-API接口設計.md) 中「五、靜態生成專用接口」保持一致,開發者可直接參考實施。

## 一、SSG 架構設計

### 1.1 整體架構

靜態生成層是本項目前台頁面的唯一渲染來源。後端 Worker 為純 API 服務,不承擔任何頁面渲染職責。SSG 服務通過調用 Worker 提供的專用接口拉取全站數據,在本地渲染為靜態 HTML 文件,再同步至 CDN 供終端用戶訪問。

```
┌─────────────────────────────────────────────────────────────────┐
│                         終端用戶訪問層                            │
│   瀏覽器/爬蟲 ──→ CDN（Cloudflare Pages / R2 公開訪問）           │
│                    ──→ 靜態 HTML 文件                             │
└─────────────────────────────────────────────────────────────────┘
        ▲ 同步靜態文件
        │
┌───────┴─────────────────────────────────────────────────────────┐
│                    靜態生成層（香港自有伺服器）                     │
│                                                                  │
│  ┌──────────┐   ┌────────────┐   ┌──────────┐   ┌───────────┐  │
│  │ 數據拉取  │──→│ 模板渲染    │──→│ 文件輸出  │──→│ CDN 同步   │  │
│  │ HTTP API │   │ Go html/tpl│   │ 本地磁碟  │   │ R2/rsync  │  │
│  └──────────┘   └────────────┘   └──────────┘   └───────────┘  │
│        │                                                         │
│        │ 配置文件                                                 │
│        ▼                                                         │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────────────────┐ │
│  │ ssg.yaml     │  │ templates/    │  │ .ssg-state.json      │ │
│  │ API地址/密鑰  │  │ 模板文件       │  │ 生成進度/時間戳        │ │
│  └──────────────┘  └───────────────┘  └──────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
        │ HTTP 拉取數據
        ▼
┌─────────────────────────────────────────────────────────────────┐
│                    後端 API 層（Cloudflare Workers）               │
│   Rust + workers-rs  →  D1（數據） / KV（緩存） / R2（存儲）       │
│   /api/v1/ssg/*  專用接口                                         │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 部署拓撲

| 節點 | 位置 | 角色 | 說明 |
|------|------|------|------|
| Worker API | Cloudflare 邊緣網絡 | 數據源 | 提供 SSG 專用 JSON 接口 |
| SSG 服務 | 香港自有伺服器 | 生成器 | Go 程序,定時或手動觸發生成 |
| 靜態文件存儲 | Cloudflare R2 | 對象存儲 | 存放生成的 HTML 文件 |
| CDN 分發 | Cloudflare Pages / R2 公開訪問 | 邊緣緩存 | 用戶直接訪問靜態頁面 |
| 管理後台 | Cloudflare Pages | SPA | 編輯內容後觸發 SSG 重新生成 |

**數據流向:**

```
管理員在後台發布/修改內容
  → Worker 寫入 D1
  → 管理後台觸發 SSG Webhook（或定時任務輪詢）
  → SSG 服務調用 /api/v1/ssg/incremental 拉取變更
  → 重新生成受影響頁面
  → 同步至 R2 / CDN
  → 用戶訪問到最新頁面
```

### 1.3 兩種生成模式

| 模式 | 觸發場景 | 拉取範圍 | 耗時 | 對 API 的壓力 |
|------|---------|---------|------|-------------|
| **全量生成** | 首次部署、大改版、模板升級 | 全站所有數據 | 較長（視文章數量） | 較高,分批拉取緩解 |
| **增量更新** | 日常內容發布/修改/刪除 | 僅變更部分 | 短（秒級到分鐘級） | 低,僅拉取差量 |

**全量生成**會重新生成所有頁面（首頁、所有欄目列表頁及分頁、所有文章詳情頁、所有單頁、sitemap.xml、robots.txt）,適用於首次部署或模板大改後的完整重建。

**增量更新**基於時間戳對比,僅重新生成發生變更的文章/欄目/單頁對應的頁面,並更新 sitemap。適用於日常運營場景,是最高頻的生成模式。

### 1.4 生成器技術棧選型

| 候選方案 | 語言 | 模板引擎 | 推薦度 | 理由 |
|---------|------|---------|--------|------|
| **Go** | Go | `html/template` | **首選** | 與 pbootcms-go 技術棧一致,可復用模板標籤邏輯;編譯為單一二進制,部署簡單;原生並發支持優秀 |
| Node.js | TypeScript | Nunjucks / Handlebars | 備選 | 生態豐富,模板引擎成熟;但運行時依賴 Node 環境 |
| Python | Python | Jinja2 | 備選 | 開發效率高,Jinja2 模板表達力強;但部署需 Python 環境 |

**推薦採用 Go 實現**,核心理由如下:

1. **模板邏輯可復用**:pbootcms-go 已實現 PbootCMS 的 `{pboot:xxx}` 標籤解析器和 URL 生成邏輯,可直接移植
2. **部署簡單**:`go build` 產出單一靜態二進制,無運行時依賴,適合伺服器部署
3. **並發性能**:Goroutine + Channel 天然適合 worker pool 並發生成頁面
4. **跨平台編譯**:可在開發機交叉編譯後直接上傳至香港伺服器

### 1.5 鑑權與安全

SSG 接口採用獨立的 API Key 驗證機制,與管理員 JWT Token 體系完全隔離。

**請求頭格式:**

```
X-SSG-Key: <ssg_api_key>
```

**安全策略:**

| 策略 | 說明 |
|------|------|
| API Key 驗證 | SSG 服務在每次請求中攜帶 `X-SSG-Key` 頭,Worker 校驗是否匹配 `ay_config` 中的 `api_token` 值 |
| IP 白名單（可選） | Worker 可配置只允許香港伺服器 IP 訪問 `/api/v1/ssg/*` 路由 |
| HTTPS 傳輸 | 所有 API 請求走 HTTPS,API Key 不會明文傳輸 |
| Key 輪換 | API Key 存儲在 `ay_config` 的 `api_token` 配置項中,可在後台隨時修改 |

> **配置位置:** API Key 對應 `ay_config` 表中 `name = 'api_token'` 的記錄（見 [02-數據庫設計.md](./02-數據庫設計.md) 5.4 節,序號 39）。SSG 服務的 `ssg.yaml` 配置文件中填寫相同的值。

## 二、全量生成流程

全量生成用於首次部署、大改版或模板升級後的完整重建。以下是完整的執行流程:

```
┌─────────────┐
│  開始全量生成 │
└──────┬──────┘
       │
       ▼
┌──────────────────────────────────────────┐
│ Step 1: 拉取全站配置                        │
│ GET /api/v1/ssg/config                    │
│ → site, company, config, nav,             │
│   slides, links, tags, labels             │
│ → 緩存為全局模板變量                         │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 2: 拉取欄目樹                          │
│ GET /api/v1/sorts?status=1                │
│ GET /api/v1/nav                            │
│ → 構建欄目樹,計算每個欄目的 URL               │
│ → 識別列表型欄目(type=1)和單頁型欄目(type=2)  │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 3: 拉取所有單頁                        │
│ GET /api/v1/singles                       │
│ → 獲取所有單頁內容（關於我們、聯繫我們等）     │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 4: 分批拉取所有文章基礎信息              │
│ GET /api/v1/ssg/contents/batch            │
│   ?page=1&pagesize=100                    │
│   &include_content=0                      │
│ → 循環翻頁直到 total_pages                  │
│ → 收集所有文章 ID 列表                       │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 5: 拉取每篇文章完整詳情                  │
│ GET /api/v1/ssg/contents/:id/full         │
│ → 並發拉取（worker pool, 限制並發數）         │
│ → 每篇含: content, sort, prev, next,       │
│   related, images, tags_data              │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 6: 生成所有靜態頁面                     │
│  ├─ 首頁 index.html                        │
│  ├─ 每個列表欄目的列表頁 + 分頁               │
│  │   /{sort_filename}/index.html          │
│  │   /{sort_filename}_{page}.html         │
│  ├─ 每篇文章的詳情頁                         │
│  │   /{sort_filename}/{id}.html           │
│  ├─ 每個單頁                               │
│  │   /{filename}.html                      │
│  ├─ sitemap.xml                            │
│  └─ robots.txt                             │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 7: 同步至 CDN                          │
│ → 上傳至 Cloudflare R2 / rsync 推送         │
│ → 清除 CDN 舊緩存                           │
│ → 記錄本次生成時間戳到 .ssg-state.json      │
└──────────────────┬───────────────────────┘
                   │
                   ▼
              ┌────────────┐
              │  生成完成    │
              └────────────┘
```

**流程詳述:**

1. **拉取全站配置** — 調用 `GET /api/v1/ssg/config` 一次性獲取站點信息、公司信息、系統配置、導航樹、幻燈片、友情連結、標籤、自定義標籤等全局數據。這些數據在所有頁面模板中共享,只需拉取一次,緩存在內存中供後續渲染使用。

2. **拉取欄目樹** — 調用 `GET /api/v1/sorts?status=1` 獲取所有啟用欄目的平鋪列表,再調用 `GET /api/v1/nav` 獲取遞歸構建的樹形導航結構。根據欄目的 `mcode`（模型編碼）和 `type` 字段區分列表型欄目（`type=1`,需生成列表頁+文章頁）和單頁型欄目（`type=2`,生成單頁）。同時根據 URL 路由規則（見第五節）為每個欄目計算其靜態文件路徑。

3. **拉取所有單頁** — 調用 `GET /api/v1/singles` 獲取所有單頁內容,與單頁型欄目按 `scode` 關聯,準備渲染單頁 HTML。

4. **分批拉取所有文章基礎信息** — 調用 `GET /api/v1/ssg/contents/batch?page=N&pagesize=100&include_content=0` 分頁拉取。設 `include_content=0` 先只拉取基礎信息（不含正文）,用於構建文章 ID 清單和列表頁數據。循環翻頁直到 `meta.total_pages`。此步驟避免單次請求返回過大數據量。

5. **拉取每篇文章完整詳情** — 對 Step 4 收集的每個文章 ID,調用 `GET /api/v1/ssg/contents/:id/full` 獲取完整渲染數據（含正文、欄目信息、上下篇、相關文章、圖片列表、標籤數據）。使用 worker pool 並發拉取,建議並發數控制在 5~10 之間,避免對 Worker API 造成瞬時壓力。

6. **生成所有靜態頁面** — 依次渲染各類頁面:
   - **首頁**:使用首頁模板,注入最新文章、推薦文章、幻燈片、友情連結等數據
   - **欄目列表頁**:每個列表型欄目按分頁配置生成首頁和各分頁
   - **文章詳情頁**:每篇文章使用其欄目指定的 `contenttpl` 模板渲染
   - **單頁**:每個單頁使用單頁模板渲染
   - **sitemap.xml**:匯總所有頁面 URL,生成標準 sitemap
   - **robots.txt**:根據站點配置生成,包含 sitemap 地址

7. **同步至 CDN** — 將生成的靜態文件同步至 CDN（推薦上傳至 Cloudflare R2,通過 R2 + Pages 實現分發）。同步完成後清除 CDN 舊緩存,確保用戶訪問到最新頁面。最後將當前時間戳寫入 `.ssg-state.json`,作為下次增量更新的基準。

## 三、增量更新流程

增量更新是日常運營的主要生成模式。基於上次同步的時間戳,僅拉取和重新生成發生變更的頁面。

```
┌──────────────────────────────────────────┐
│ Step 1: 讀取上次同步時間戳                   │
│ 從 .ssg-state.json 讀取 last_sync 欄位     │
│ → 例如: 1700000000                         │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 2: 調用增量接口                        │
│ GET /api/v1/ssg/incremental?since=        │
│   1700000000                               │
│ → 返回 since 之後的所有變更                  │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 3: 解析變更數據                        │
│  ├─ contents.updated → 新增/修改的文章 ID    │
│  ├─ contents.deleted → 已刪除的文章 ID       │
│  ├─ singles.updated → 新增/修改的單頁        │
│  ├─ singles.deleted → 已刪除的單頁           │
│  ├─ sorts.updated → 新增/修改的欄目          │
│  ├─ sorts.deleted → 已刪除的欄目             │
│  ├─ config_changed → 配置是否有變更          │
│  ├─ site_changed → 站點信息是否有變更         │
│  ├─ company_changed → 公司信息是否有變更      │
│  ├─ slides_changed → 幻燈片是否有變更         │
│  └─ links_changed → 友情連結是否有變更        │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 4: 只重新生成受影響的頁面               │
│                                            │
│ 4a. 配置/站點/公司/幻燈片/連結變更時:        │
│     → 重新拉取 GET /api/v1/ssg/config      │
│     → 更新全局模板變量                       │
│     → 觸發全站頁面重新生成（降級為全量）       │
│                                            │
│ 4b. 欄目變更時:                             │
│     → 重新拉取欄目樹                         │
│     → 重新生成受影響欄目的列表頁             │
│     → 重新生成該欄目下所有文章頁              │
│     → 更新導航（影響所有頁面頭部導航）         │
│                                            │
│ 4c. 文章新增/修改時:                         │
│     → GET /api/v1/ssg/contents/:id/full    │
│     → 重新生成該文章詳情頁                   │
│     → 重新生成其所在欄目的列表頁              │
│     → 重新生成其上下篇文章頁（prev/next 變化）│
│                                            │
│ 4d. 文章刪除時:                             │
│     → 刪除對應的靜態 HTML 文件               │
│     → 重新生成其所在欄目的列表頁              │
│     → 重新生成其上下篇文章頁                  │
│                                            │
│ 4e. 單頁新增/修改時:                         │
│     → 重新生成對應單頁                       │
│                                            │
│ 4f. 單頁刪除時:                             │
│     → 刪除對應靜態文件                       │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 5: 更新 sitemap                        │
│ GET /api/v1/ssg/sitemap?type=xml           │
│ → 重新生成 sitemap.xml                      │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Step 6: 同步至 CDN                          │
│ → 僅同步變更的文件                           │
│ → 清除對應 CDN 緩存                          │
│ → 更新 .ssg-state.json 的 last_sync        │
└──────────────────┬───────────────────────┘
                   │
                   ▼
              ┌────────────┐
              │  增量完成    │
              └────────────┘
```

**增量更新策略要點:**

| 變更類型 | 影響範圍 | 生成動作 |
|---------|---------|---------|
| 配置/站點/公司/幻燈片/連結變更 | 全站 | 降級為全量重新生成（全局變量變化影響所有頁面） |
| 欄目新增/修改 | 該欄目 + 導航 | 重新生成該欄目列表頁及其文章頁;因導航變化需重新生成所有頁面頭部 |
| 欄目刪除 | 該欄目 | 刪除該欄目下所有靜態文件;重新生成導航相關頁面 |
| 文章新增/修改 | 該文章 + 所在欄目列表 + 上下篇 | 重新生成文章頁、欄目列表頁、上下篇文章頁 |
| 文章刪除 | 該文章 + 所在欄目列表 + 上下篇 | 刪除文章頁;重新生成欄目列表頁、上下篇文章頁 |
| 單頁新增/修改 | 該單頁 | 重新生成單頁 |
| 單頁刪除 | 該單頁 | 刪除單頁靜態文件 |

> **重要:** 當 `config_changed`、`site_changed`、`company_changed`、`slides_changed` 或 `links_changed` 任一為 `true` 時,由於全局模板變量發生變化（站點標題、Logo、聯繫方式、幻燈片等出現在每個頁面），應降級為全量重新生成。這是保證頁面一致性的必要措施。

> **時間戳基準:** `.ssg-state.json` 中的 `last_sync` 應使用增量接口返回的 `until` 值（即本次增量查詢的截止時間）,而非本地當前時間。這樣即使生成耗時較長,也不會漏掉在生成期間發生的變更。

**字段說明:**

| 字段 | 類型 | 說明 |
|------|------|------|
| `since` | `number` | 查詢起始時間戳（即請求參數 `since` 的回顯） |
| `until` | `number` | 查詢截止時間戳（服務端當前時間）。SSG 服務應將此值保存為新的 `last_sync` |
| `contents.updated` | `array` | 新增或修改的文章列表（含基礎字段） |
| `contents.deleted` | `array` | 被刪除的文章 ID 數組 |
| `singles.updated` | `array` | 新增或修改的單頁列表 |

| `site_changed` | `boolean` | 站點信息是否有變更 |
| `company_changed` | `boolean` | 公司信息是否有變更 |
| `slides_changed` | `boolean` | 幻燈片是否有變更 |
| `links_changed` | `boolean` | 友情連結是否有變更 |

**增量更新決策邏輯:**

```
收到增量響應後:

IF config_changed OR site_changed OR company_changed
   OR slides_changed OR links_changed:
    → 全局變量變化,降級為全量重新生成
    → 調用 /ssg/config 刷新全局數據

ELSE:
    FOR EACH sort IN sorts.updated:
        → 重新拉取欄目樹,更新內存中的欄目結構
        → 重新生成該欄目列表頁及其下所有文章頁

    FOR EACH sort_id IN sorts.deleted:
        → 刪除該欄目對應的靜態文件目錄
        → 重新拉取欄目樹更新導航

    FOR EACH content IN contents.updated:
        → 調用 /ssg/contents/:id/full 獲取完整數據
        → 重新生成該文章詳情頁
        → 重新生成其所在欄目的列表頁（文章列表變化）
        → 重新生成其上下篇文章頁（導航鏈接變化）

    FOR EACH content_id IN contents.deleted:
        → 刪除該文章的靜態 HTML 文件
        → 重新生成其所在欄目的列表頁
        → 重新生成其上下篇文章頁

    FOR EACH single IN singles.updated:
        → 重新生成對應單頁

    FOR EACH single_id IN singles.deleted:
        → 刪除對應單頁靜態文件

    → 重新生成 sitemap.xml
    → 同步變更文件至 CDN

SAVE until → .ssg-state.json.last_sync
```

### 4.6 GET /api/v1/ssg/sitemap

生成 sitemap 數據,用於搜索引擎收錄。返回所有需要收錄的 URL 列表。支持 JSON 和 XML 兩種返回格式。

**請求:**

```
GET /api/v1/ssg/sitemap?type=xml
X-SSG-Key: <ssg_api_key>
```

**Query 參數:**

| 參數 | 類型 | 必填 | 默認值 | 說明 |
|------|------|------|--------|------|
| `type` | `string` | 否 | `json` | 返回格式:`json` 或 `xml` |

**響應（type=json 時）`data` 結構:**

```json
{
  "urls": [
    {
      "loc": "https://example.com/",
      "lastmod": "2024-01-01",
      "changefreq": "daily",
      "priority": "1.0"
    },
    {
      "loc": "https://example.com/news/",
      "lastmod": "2024-01-01",
      "changefreq": "daily",
      "priority": "0.9"
    },
    {
      "loc": "https://example.com/news/101.html",
      "lastmod": "2024-01-01",
      "changefreq": "weekly",
      "priority": "0.8"
    },
    {
      "loc": "https://example.com/about.html",
      "lastmod": "2024-01-01",
      "changefreq": "monthly",
      "priority": "0.7"
    }
  ],
  "total": 500
}
```

**響應（type=xml 時）直接返回 XML:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://example.com/</loc>
    <lastmod>2024-01-01</lastmod>
    <changefreq>daily</changefreq>
    <priority>1.0</priority>
  </url>
  <url>
    <loc>https://example.com/news/</loc>
    <lastmod>2024-01-01</lastmod>
    <changefreq>daily</changefreq>
    <priority>0.9</priority>
  </url>
</urlset>
```

**URL 優先級規則:**

| 頁面類型 | `priority` | `changefreq` | 說明 |
|---------|-----------|-------------|------|
| 首頁 | `1.0` | `daily` | 最高優先級 |
| 欄目列表頁 | `0.9` | `daily` | 高優先級 |
| 文章詳情頁 | `0.8` | `weekly` | 中等優先級 |
| 單頁 | `0.7` | `monthly` | 低頻更新 |

> SSG 服務可直接調用 `type=xml` 獲取 XML 格式並保存為 `sitemap.xml`。也可調用 `type=json` 獲取結構化數據後自行處理（如分拆多個 sitemap 文件）。

## 五、URL 路由規則

### 5.1 三種 URL 模式

參考 PbootCMS 的 URL 生成邏輯,系統支持三種 URL 模式,通過 `ay_config` 表中 `url_type` 配置項控制:

| `url_type` 值 | 模式名稱 | URL 格式示例 | 說明 |
|--------------|---------|-------------|------|
| `1` | 普通模式 | `/index.php?c=list&scode=1001` | 帶查詢參數的傳統 URL,適合不支持偽靜態的環境 |
| `2` | **偽靜態模式** | `/news/`、`/news/101.html` | 簡潔的偽靜態 URL,**默認使用此模式** |
| `3` | 兼容模式 | `/index.php/news/101.html` | 兼容模式,在入口文件後帶路徑,適合不支持 URL 重寫的環境 |

> **SSG 場景下的選擇:** 由於 SSG 直接生成靜態 HTML 文件,普通模式（`url_type=1`）和兼容模式（`url_type=3`）依賴動態入口文件,不適合純靜態部署。因此 SSG 服務固定使用**偽靜態模式（`url_type=2`）**生成靜態文件路徑。

### 5.2 偽靜態模式 URL 規則（默認）

偽靜態模式下,URL 規則由 `ay_config` 中的以下配置項決定:

| 配置項 | 默認值 | 說明 |
|--------|--------|------|
| `url_sort` | `{filename}` | 欄目 URL 規則 |
| `url_content` | `{scode}/{filename}` | 文章 URL 規則 |
| `url_tag` | `tag/{tag}` | 標籤 URL 規則 |

**佔位符說明:**

| 佔位符 | 含義 | 取值來源 |
|--------|------|---------|
| `{filename}` | 欄目/文章的 URL 別名 | `ay_content_sort.filename` 或 `ay_content.filename` |
| `{scode}` | 欄目編碼 | `ay_content_sort.scode` |
| `{urlname}` | URL 別名（備用） | `ay_content_sort.urlname` 或 `ay_content.urlname` |
| `{id}` | 文章 ID | `ay_content.id` |
| `{tag}` | 標籤名 | 標籤文字 |

**各頁面類型 URL 規則:**

| 頁面類型 | URL 規則 | 靜態文件路徑 | 示例 |
|---------|---------|-------------|------|
| 欄目列表頁（首頁） | `/{filename}` 或 `/{urlname}_{scode}` | `/{sort_filename}/index.html` | `/news/` → `news/index.html` |
| 文章詳情頁 | `/{sort_filename}/{content_filename}.html` 或 `/{sort_filename}/{id}.html` | `/{sort_filename}/{id}.html` | `/news/101.html` |
| 單頁 | `/{filename}` 或 `/{urlname}_{scode}` | `/{filename}.html` | `/about.html` |
| 欄目分頁 | `/{filename}_{page}.html` | `/{sort_filename}_{page}.html` | `/news_2.html` |
| 標籤頁 | `/tag/{tag}` | `/tag/{tag}.html` | `/tag/SEO優化.html` |

> **文章 URL 的兩種形式:** 當文章設置了 `filename`（URL 別名）時,使用 `/{sort_filename}/{content_filename}.html`;當未設置 `filename` 時,使用 `/{sort_filename}/{id}.html`。SSG 服務應優先使用 `filename`,為空時回退到 `id`。

> **欄目 URL 的兩種形式:** 當欄目設置了 `filename` 時,使用 `/{filename}`;當未設置 `filename` 時,使用 `/{urlname}_{scode}`（URL 別名 + 欄目編碼,確保唯一性）。

### 5.3 URL 生成示例

假設有以下欄目和文章數據:

| 欄目 scode | 欄目名稱 | filename | urlname | type |
|-----------|---------|----------|---------|------|
| `1001` | 公司新聞 | `news` | `company-news` | 1（列表） |
| `1002` | 行業動態 | `industry` | `industry-news` | 1（列表） |
| `2001` | 關於我們 | `about` | `about` | 2（單頁） |
| `2002` | 聯繫我們 | `contact` | `contact` | 2（單頁） |

| 文章 ID | scode | title | filename |
|---------|-------|-------|----------|
| `101` | `1001` | 公司獲得行業大獎 | （空） |
| `102` | `1001` | 年度總結大會 | `annual-summary` |
| `103` | `1002` | 行業最新趨勢 | （空） |

**生成的 URL 和靜態文件路徑:**

| 頁面 | URL | 靜態文件路徑 |
|------|-----|-------------|
| 首頁 | `/` | `index.html` |
| 公司新聞列表（第1頁） | `/news/` | `news/index.html` |
| 公司新聞列表（第2頁） | `/news_2.html` | `news_2.html` |
| 行業動態列表 | `/industry/` | `industry/index.html` |
| 文章101詳情 | `/news/101.html` | `news/101.html` |
| 文章102詳情 | `/news/annual-summary.html` | `news/annual-summary.html` |
| 文章103詳情 | `/industry/103.html` | `industry/103.html` |
| 關於我們（單頁） | `/about.html` | `about.html` |
| 聯繫我們（單頁） | `/contact.html` | `contact.html` |
| sitemap | `/sitemap.xml` | `sitemap.xml` |
| robots | `/robots.txt` | `robots.txt` |

### 5.4 分頁 URL 規則

欄目列表頁分頁 URL 格式為 `/{filename}_{page}.html`,其中 `{page}` 從 2 開始（第 1 頁為欄目首頁 `/{filename}/` 或 `/{filename}/index.html`）。

| 頁碼 | URL | 靜態文件路徑 |
|------|-----|-------------|
| 第 1 頁 | `/news/` | `news/index.html` |
| 第 2 頁 | `/news_2.html` | `news_2.html` |
| 第 3 頁 | `/news_3.html` | `news_3.html` |
| 第 N 頁 | `/news_N.html` | `news_N.html` |

**分頁計算邏輯:**

```
每頁條數 = config.page_num（默認 10）
總文章數 = 該欄目下文章總數
總頁數 = ceil(總文章數 / 每頁條數)

FOR page = 1 TO 總頁數:
    IF page == 1:
        輸出路徑 = {sort_filename}/index.html
    ELSE:
        輸出路徑 = {sort_filename}_{page}.html
    渲染列表頁模板,注入當前頁文章數據和分頁導航
```

> **分頁配置:** 每頁條數由 `ay_config` 的 `page_num` 控制（默認 `10`）。分頁顯示文字由 `page_word` 控制（默認 `首頁,上一頁,下一頁,尾頁`）。

### 7.2 HTTP 客戶端與重試

SSG 服務通過 HTTP 客戶端調用 Worker API,需實現自動重試機制:

```go
package client

import (
    "context"
    "encoding/json"
    "fmt"
    "io"
    "net/http"
    "time"
)

type APIClient struct {
    baseURL    string
    apiKey     string
    httpClient *http.Client
    retryCount int
    retryDelay time.Duration
}

func NewAPIClient(cfg Config) *APIClient {
    return &APIClient{
        baseURL: cfg.BaseURL,
        apiKey:  cfg.APIKey,
        httpClient: &http.Client{
            Timeout: cfg.Timeout,
        },
        retryCount: cfg.RetryCount,
        retryDelay: cfg.RetryDelay,
    }
}

// Get 帶自動重試的 GET 請求
func (c *APIClient) Get(ctx context.Context, path string, params url.Values) (*APIResponse, error) {
    var lastErr error
    
    for attempt := 0; attempt <= c.retryCount; attempt++ {
        if attempt > 0 {
            // 指數退避: 第1次重試等2s,第2次等4s,第3次等8s
            delay := c.retryDelay * time.Duration(1<<(attempt-1))
            select {
            case <-time.After(delay):
            case <-ctx.Done():
                return nil, ctx.Err()
            }
        }
        
        resp, err := c.doGet(ctx, path, params)
        if err == nil && resp.Code == 0 {
            return resp, nil
        }
        
        lastErr = fmt.Errorf("attempt %d: %w", attempt+1, err)
        if resp != nil && resp.Code != 0 {
            lastErr = fmt.Errorf("attempt %d: API error code=%d msg=%s",
                attempt+1, resp.Code, resp.Msg)
        }
    }
    
    return nil, lastErr
}

func (c *APIClient) doGet(ctx context.Context, path string, params url.Values) (*APIResponse, error) {
    u := c.baseURL + path
    if len(params) > 0 {
        u += "?" + params.Encode()
    }
    
    req, err := http.NewRequestWithContext(ctx, "GET", u, nil)
    if err != nil {
        return nil, err
    }
    req.Header.Set("X-SSG-Key", c.apiKey)
    req.Header.Set("Accept", "application/json")
    
    resp, err := c.httpClient.Do(req)
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()
    
    body, err := io.ReadAll(resp.Body)
    if err != nil {
        return nil, err
    }
    
    var result APIResponse
    if err := json.Unmarshal(body, &result); err != nil {
        return nil, fmt.Errorf("JSON unmarshal failed: %w", err)
    }
    
    return &result, nil
}

// APIResponse 對應統一響應格式
type APIResponse struct {
    Code int             `json:"code"`
    Msg  string          `json:"msg"`
    Data json.RawMessage `json:"data"`
    Meta *PaginationMeta `json:"meta,omitempty"`
}

type PaginationMeta struct {
    Page       int `json:"page"`
    Pagesize   int `json:"pagesize"`
    Total      int `json:"total"`
    TotalPages int `json:"total_pages"`
}
```

### 7.3 並發生成（Worker Pool）

使用 worker pool 並發拉取文章詳情和生成頁面,控制並發數避免壓垮 API:

```go
package generator

import (
    "context"
    "sync"
)

// WorkerPool 並發任務池
type WorkerPool struct {
    concurrency int
    tasks       chan Task
    wg          sync.WaitGroup
    errCh       chan error
}

type Task func() error

func NewWorkerPool(concurrency int) *WorkerPool {
    return &WorkerPool{
        concurrency: concurrency,
        tasks:       make(chan Task, concurrency*2),
        errCh:       make(chan error, 1),
    }
}

func (p *WorkerPool) Run(ctx context.Context) {
    for i := 0; i < p.concurrency; i++ {
        p.wg.Add(1)
        go p.worker(ctx)
    }
}

func (p *WorkerPool) worker(ctx context.Context) {
    defer p.wg.Done()
    for {
        select {
        case <-ctx.Done():
            return
        case task, ok := <-p.tasks:
            if !ok {
                return
            }
            if err := task(); err != nil {
                select {
                case p.errCh <- err:
                default: // 只記錄第一個錯誤
                }
            }
        }
    }
}

func (p *WorkerPool) Submit(task Task) {
    p.tasks <- task
}

func (p *WorkerPool) Wait() error {
    close(p.tasks)
    p.wg.Wait()
    close(p.errCh)
    return <-p.errCh
}

// 使用示例: 並發拉取文章詳情並生成頁面
func (g *Generator) generateContentPages(ctx context.Context, contentIDs []int) error {
    pool := NewWorkerPool(g.cfg.Concurrency)
    pool.Run(ctx)
    
    for _, id := range contentIDs {
        id := id // 閉包捕獲
        pool.Submit(func() error {
            // 1. 拉取文章完整詳情
            full, err := g.client.GetContentFull(ctx, id)
            if err != nil {
                return fmt.Errorf("content %d: %w", id, err)
            }
            
            // 2. 渲染模板
            html, err := g.render.RenderContent(full)
            if err != nil {
                return fmt.Errorf("render content %d: %w", id, err)
            }
            
            // 3. 寫入文件
            path := g.urlBuilder.ContentFilePath(full.Sort.Filename, id, full.Content.Filename)
            return g.writeFile(path, html)
        })
    }
    
    return pool.Wait()
}
```

### 7.4 模板引擎

推薦使用 Go 標準庫 `html/template`,理由:

1. 標準庫無需額外依賴,編譯後二進制更小
2. 自帶 HTML 轉義,天然防 XSS
3. 支持模板嵌套（`{{template "name" .}}`）,可實現佈局繼承
4. 支持自定義函數,可實現 PbootCMS 標籤的等價功能

**模板輔助函數示例:**

```go
package render

import (
    "html/template"
    "strings"
    "time"
)

func NewTemplateEngine(templateDir string) (*template.Template, error) {
    funcMap := template.FuncMap{
        // 日期格式化
        "formatDate": func(ts int64) string {
            return time.Unix(ts, 0).Format("2006-01-02")
        },
        "formatDateTime": func(ts int64) string {
            return time.Unix(ts, 0).Format("2006-01-02 15:04:05")
        },
        // 截取摘要
        "truncate": func(s string, n int) string {
            runes := []rune(s)
            if len(runes) <= n {
                return s
            }
            return string(runes[:n]) + "..."
        },
        // 字符串包含
        "contains": strings.Contains,
        // 條件輸出
        "ternary": func(cond bool, a, b interface{}) interface{} {
            if cond {
                return a
            }
            return b
        },
        // 內鏈替換（標籤自動加鏈）
        "autoLink": func(content string, tags []TagData) template.HTML {
            result := content
            for _, tag := range tags {
                link := fmt.Sprintf(`<a href="%s">%s</a>`, tag.Link, tag.Name)
                result = strings.Replace(result, tag.Name, link, 1) // 只替換第一個
            }
            return template.HTML(result) // 已轉義的 HTML
        },
    }
    
    return template.New("").Funcs(funcMap).ParseGlob(templateDir + "/*.html")
}
```

  "generated_sort_codes": ["1001", "1002"],

**斷點續傳邏輯:**

```go
package state

import (
    "encoding/json"
    "os"
    "sync"
)

type State struct {
    LastSync          int64  `json:"last_sync"`
    LastFullBuild     int64  `json:"last_full_build"`
    LastBuildType     string `json:"last_build_type"`
    GeneratedContents []int  `json:"generated_content_ids"`
    GeneratedSorts    []string `json:"generated_sort_codes"`
    GeneratedSingles  []int  `json:"generated_single_ids"`
    PendingContents   []int  `json:"pending_content_ids"`
    BuildStatus       string `json:"build_status"` // running | completed | interrupted
    BuildStartedAt    int64  `json:"build_started_at"`
    BuildFinishedAt   int64  `json:"build_finished_at"`
    
    mu   sync.Mutex
    path string
}

func Load(path string) (*State, error) {
    s := &State{path: path}
    data, err := os.ReadFile(path)
    if err != nil {
        if os.IsNotExist(err) {
            return s, nil // 首次運行,返回空狀態
        }
        return nil, err
    }
    return s, json.Unmarshal(data, s)
}

func (s *State) Save() error {
    s.mu.Lock()
    defer s.mu.Unlock()
    
    data, err := json.MarshalIndent(s, "", "  ")
    if err != nil {
        return err
    }
    return os.WriteFile(s.path, data, 0644)
}

// MarkContentGenerated 標記某篇文章已生成
func (s *State) MarkContentGenerated(id int) {
    s.mu.Lock()
    defer s.mu.Unlock()
    s.GeneratedContents = append(s.GeneratedContents, id)
}

// IsContentGenerated 檢查某篇文章是否已生成（用於斷點續傳）
func (s *State) IsContentGenerated(id int) bool {
    s.mu.Lock()
    defer s.mu.Unlock()
    for _, gid := range s.GeneratedContents {
        if gid == id {
            return true
        }
    }
    return false
}

// ResumeCheck 檢查上次生成是否中斷,返回需要繼續的文章 ID
func (s *State) ResumeCheck(allContentIDs []int) []int {
    if s.BuildStatus != "interrupted" {
        return allContentIDs
    }
    
    // 過濾出尚未生成的文章 ID
    generated := make(map[int]bool)
    for _, id := range s.GeneratedContents {
        generated[id] = true
    }
    
    var pending []int
    for _, id := range allContentIDs {
        if !generated[id] {
            pending = append(pending, id)
        }
    }
    return pending
}
```

## 八、CDN 同步策略

### 8.1 同步方式對比

| 同步方式 | 延遲 | 可靠性 | 成本 | 適用場景 |
|---------|------|--------|------|---------|
| **R2 + Pages** | 低 | 高 | R2 存儲費 + Pages 免費 | **推薦**,與 Cloudflare 生態無縫集成 |
| rsync 直推 | 中 | 中 | 伺服器帶寬成本 | 自有 CDN 伺服器場景 |
| SFTP 上傳 | 中 | 中 | 伺服器帶寬成本 | 簡單場景 |
| Webhook 觸發構建 | 較高 | 高 | Pages 構建時間 | 使用 Pages 原生構建流程 |

### 8.2 R2 + Pages 方案（推薦）

生成完成後,將靜態文件上傳至 Cloudflare R2 存儲桶,通過 R2 公開訪問或綁定 Pages 項目實現 CDN 分發。

**架構:**

```
SSG 服務生成 HTML
    │
    ├──→ 上傳至 R2 存儲桶（rust-cms-static）
    │        │
    │        ├──→ 方式A: R2 自定義域名公開訪問
    │        │    https://static.example.com/index.html
    │        │
    │        └──→ 方式B: R2 綁定 Pages 項目
    │             https://www.example.com/index.html
    │
    └──→ 觸發 CDN 緩存清理
```

**R2 上傳實現:**

```go
package sync

import (
    "context"
    "os"
    "path/filepath"
    
    "github.com/aws/aws-sdk-go-v2/config"
    "github.com/aws/aws-sdk-go-v2/service/s3"
)

type R2Sync struct {
    client   *s3.Client
    bucket   string
    publicURL string
}

func NewR2Sync(cfg R2Config) (*R2Sync, error) {
    // R2 兼容 S3 API,使用 AWS SDK 配置
    r2Config, err := config.LoadDefaultConfig(context.Background(),
        config.WithCredentialsProvider(credentials.NewStaticCredentialsProvider(
            cfg.AccessKeyID,
            cfg.SecretAccessKey,
            "",
        )),
        config.WithRegion("auto"),
    )
    if err != nil {
        return nil, err
    }
    
    client := s3.NewFromConfig(r2Config, func(o *s3.Options) {
        o.BaseEndpoint = aws.String(
            fmt.Sprintf("https://%s.r2.cloudflarestorage.com", cfg.AccountID),
        )
    })
    
    return &R2Sync{
        client:    client,
        bucket:    cfg.Bucket,
        publicURL: cfg.PublicURL,
    }, nil
}

// SyncDir 同步整個輸出目錄到 R2
func (r *R2Sync) SyncDir(ctx context.Context, localDir string) error {
    return filepath.Walk(localDir, func(path string, info os.FileInfo, err error) error {
        if err != nil {
            return err
        }
        if info.IsDir() {
            return nil
        }
        
        // 計算 R2 對象 key（相對路徑）
        relPath, err := filepath.Rel(localDir, path)
        if err != nil {
            return err
        }
        // 統一為 URL 路徑格式
        key := filepath.ToSlash(relPath)
        
        return r.uploadFile(ctx, path, key, info)
    })
}

// SyncFiles 僅同步變更文件（增量更新用）
func (r *R2Sync) SyncFiles(ctx context.Context, localDir string, changedFiles []string) error {
    for _, relFile := range changedFiles {
        localPath := filepath.Join(localDir, relFile)
        key := filepath.ToSlash(relFile)
        
        if err := r.uploadFile(ctx, localPath, key, nil); err != nil {
            return err
        }
    }
    return nil
}

func (r *R2Sync) uploadFile(ctx context.Context, localPath, key string, info os.FileInfo) error {
    file, err := os.Open(localPath)
    if err != nil {
        return err
    }
    defer file.Close()
    
    contentType := guessContentType(key)
    
    _, err = r.client.PutObject(ctx, &s3.PutObjectInput{
        Bucket:      aws.String(r.bucket),
        Key:         aws.String(key),
        Body:        file,
        ContentType: aws.String(contentType),
    })
    return err
}

func guessContentType(key string) string {
    switch filepath.Ext(key) {
    case ".html":
        return "text/html; charset=utf-8"
    case ".xml":
        return "application/xml; charset=utf-8"
    case ".txt":
        return "text/plain; charset=utf-8"
    case ".css":
        return "text/css; charset=utf-8"
    case ".js":
        return "application/javascript; charset=utf-8"
    case ".json":
        return "application/json; charset=utf-8"
    case ".png":
        return "image/png"
    case ".jpg", ".jpeg":
        return "image/jpeg"
    case ".gif":
        return "image/gif"
    case ".webp":
        return "image/webp"
    case ".svg":
        return "image/svg+xml"
    default:
        return "application/octet-stream"
    }
}
```

> **R2 配置要點:**
> - R2 存儲桶需開啟公開訪問或綁定自定義域名
> - 建議為 HTML 文件設置合理的 `Cache-Control` 頭（如 `public, max-age=3600`）
> - R2 兼容 S3 API,可使用 AWS SDK for Go v2 連接
> - Endpoint 格式為 `https://{account_id}.r2.cloudflarestorage.com`

### 8.3 rsync 直推方案

如果使用自有 CDN 伺服器,可通過 rsync 同步靜態文件:

```go
package sync

import (
    "os/exec"
)

type RsyncSync struct {
    target  string // user@host:/path/
    exclude string // 排除文件
}

func (r *RsyncSync) Sync(localDir string) error {
    args := []string{
        "-avz",              // 歸檔模式 + 壓縮
        "--delete",          // 刪除目標端多餘文件
        "-e", "ssh -p 22",   // SSH 端口
    }
    if r.exclude != "" {
        args = append(args, "--exclude", r.exclude)
    }
    args = append(args, localDir+"/", r.target)
    
    cmd := exec.Command("rsync", args...)
    return cmd.Run()
}
```

**rsync 命令示例:**

```bash
# 全量同步
rsync -avz --delete -e "ssh -p 22" \
    ./output/ \
    user@cdn-server:/var/www/html/

# 僅同步變更文件（增量更新,利用 rsync 的增量傳輸）
rsync -avz --delete -e "ssh -p 22" \
    --exclude=".ssg-state.json" \
    ./output/ \
    user@cdn-server:/var/www/html/
```

### 8.4 緩存清理

生成新頁面後需清除 CDN 舊緩存,確保用戶訪問到最新內容。

**Cloudflare CDN 緩存清理:**

```go
package sync

import (
    "bytes"
    "encoding/json"
    "net/http"
)

type CachePurger struct {
    zoneID   string
    apiToken string
}

// PurgeAll 清除整個 Zone 的緩存（全量生成後使用）
func (p *CachePurger) PurgeAll() error {
    body, _ := json.Marshal(map[string]interface{}{
        "purge_everything": true,
    })
    
    req, _ := http.NewRequest("POST",
        "https://api.cloudflare.com/client/v4/zones/"+p.zoneID+"/purge_cache",
        bytes.NewReader(body))
    req.Header.Set("Authorization", "Bearer "+p.apiToken)
    req.Header.Set("Content-Type", "application/json")
    
    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()
    return nil
}

// PurgeURLs 清除指定 URL 的緩存（增量更新後使用,更精準）
func (p *CachePurger) PurgeURLs(urls []string) error {
    body, _ := json.Marshal(map[string]interface{}{
        "files": urls,
    })
    
    req, _ := http.NewRequest("POST",
        "https://api.cloudflare.com/client/v4/zones/"+p.zoneID+"/purge_cache",
        bytes.NewReader(body))
    req.Header.Set("Authorization", "Bearer "+p.apiToken)
    req.Header.Set("Content-Type", "application/json")
    
    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()
    return nil
}
```

**緩存清理策略:**

| 場景 | 清理方式 | 說明 |
|------|---------|------|
| 全量生成完成 | `PurgeAll` | 清除整站緩存,確保所有頁面更新 |
| 增量更新（少量頁面） | `PurgeURLs` | 僅清除變更頁面的 URL 緩存,減少 API 調用 |
| 增量更新（大量頁面） | `PurgeAll` | 當變更 URL 超過 Cloudflare 單次限制（30 個）時,直接全量清理 |

**增量更新時構造待清理 URL 列表:**

```go
func (g *Generator) collectPurgeURLs(changes *IncrementalChanges) []string {
    var urls []string
    baseURL := g.cfg.Site.Domain
    
    // 首頁始終清除（列表變化影響首頁最新文章）
    urls = append(urls, baseURL+"/")
    
    // 變更文章的 URL
    for _, c := range changes.Contents.Updated {
        url := baseURL + g.urlBuilder.ContentURL(c.SortFilename, c.ID, c.Filename)
        urls = append(urls, url)
    }
    
    // 變更文章所在欄目的列表頁 URL
    sortURLs := make(map[string]bool)
    for _, c := range changes.Contents.Updated {
        sortURL := baseURL + "/" + c.SortFilename + "/"
        if !sortURLs[sortURL] {
            sortURLs[sortURL] = true
            urls = append(urls, sortURL)
        }
    }
    
    // 變更單頁的 URL
    for _, s := range changes.Singles.Updated {
        url := baseURL + "/" + s.Filename + ".html"
        urls = append(urls, url)
    }
    
    // sitemap
    urls = append(urls, baseURL+"/sitemap.xml")
    
    return urls
}
```

**日誌記錄建議:**

| 記錄項 | 說明 |
|--------|------|
| `build_type` | `full` 或 `incremental` |
| `duration_seconds` | 總耗時,用於監控生成效率 |
| `stats.pages_generated` | 實際生成的頁面數 |
| `stats.api_requests` | API 請求總數,監控 API 使用量 |
| `stats.api_errors` | API 錯誤數,非零表示有重試或失敗 |
| `errors` | 錯誤詳情數組,記錄所有失敗的操作 |

**監控建議:**

1. **定時生成監控**:通過 cron 定時任務（如每 5 分鐘）執行增量更新,監控生成是否正常完成
2. **Webhook 觸發監控**:管理後台發布內容後觸發 Webhook,SSG 服務收到後立即執行增量更新
3. **失敗告警**:生成失敗時通過郵件/即時通訊工具告警
4. **API 健康檢查**:定期調用 `GET /api/v1/ssg/config` 檢查 API 可達性

**Cron 定時任務示例（crontab）:**

```bash
# 每 5 分鐘執行增量更新
*/5 * * * * /opt/ssg/ssg-generator incremental --config=/etc/ssg/ssg.yaml >> /var/log/ssg/cron.log 2>&1

# 每天凌晨 3 點執行全量重建（可選,確保一致性）
0 3 * * * /opt/ssg/ssg-generator full --config=/etc/ssg/ssg.yaml >> /var/log/ssg/full.log 2>&1
```

**Webhook 觸發示例:**

SSG 服務可啟動一個輕量 HTTP 服務監聽管理後台的 Webhook 回調:

```
管理後台發布文章
  → POST https://hk-server:8080/webhook/ssg
  → SSG 服務收到後立即執行增量更新
  → 返回生成結果
```

> **Webhook 與 Cron 結合:** 建議同時使用 Webhook（即時觸發）和 Cron（兜底保障）。Webhook 確保內容發布後秒級更新,Cron 確保即使 Webhook 失敗也能在 5 分鐘內補上。
