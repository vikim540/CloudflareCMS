# 技術架構設計

> 本文件定義 Rust + Cloudflare Workers CMS 的整體技術架構,經過對 PbootCMS、pbootcms-go、AnqiCMS 三個參考項目的調研後,針對 Cloudflare 無伺服器環境重新設計。

---

## 一、架構總覽

系統分為四層,各層完全解耦,獨立部署:

```
┌─────────────────────────────────────────────────────────────┐
│                      用戶訪問層                              │
│  瀏覽器/爬蟲 ──→ CDN ──→ 靜態HTML（Cloudflare Pages/CDN）    │
│  管理員   ──→ admin.xxx.com ──→ React SPA（Cloudflare Pages）│
└─────────────────────────────────────────────────────────────┘
         │                                    │
         │ SSG拉取數據                         │ API調用
         ▼                                    ▼
┌─────────────────────┐          ┌─────────────────────────┐
│  靜態生成層          │          │  後端API層               │
│  （香港自有伺服器）   │  HTTP    │  Rust + workers-rs       │
│  Go/Node 生成服務    │ ◀──────  │  Cloudflare Workers      │
│  批量生成HTML        │          │  純API,無頁面渲染         │
└─────────────────────┘          └─────────────────────────┘
                                          │
                    ┌─────────────────────┼──────────────────┐
                    ▼                     ▼                  ▼
          ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
          │  數據存儲層    │    │  緩存層       │    │  文件存儲層   │
          │  Cloudflare  │    │  Cloudflare  │    │  Cloudflare  │
          │  D1(SQLite)  │    │  KV          │    │  R2          │
          └──────────────┘    └──────────────┘    └──────────────┘
```

### 與 PbootCMS/Go 版的架構差異

| 維度 | PbootCMS (PHP) | pbootcms-go | 本項目 (Rust+Workers) |
|------|----------------|-------------|----------------------|
| 部署方式 | 單機 LAMP/LEMP | 單機二進制 | Cloudflare 邊緣網絡 |
| 前台渲染 | PHP 模板引擎同步渲染 | Go 模板引擎同步渲染 | 無渲染,純靜態HTML |
| 後台 | PHP 頁面 | Go 模板頁面 | React SPA (獨立部署) |
| 數據庫 | MySQL/SQLite | SQLite | D1 (SQLite) |
| 緩存 | 文件緩存 | 內存 + 文件 | KV |
| 文件存儲 | 本地磁碟 | 本地 + R2 | R2 |
| 無狀態 | 否（Session文件） | 否（內存Session） | 是（JWT + KV） |

## 二、技術棧選型

### 後端 Worker

| 組件 | 選型 | 版本要求 | 選型理由 |
|------|------|---------|---------|
| SDK | `worker` crate | >= 0.8.0 | Cloudflare 官方 Rust SDK,支持 D1/KV/R2/Router |
| 序列化 | `serde` + `serde_json` | latest | Rust 生態標準,wasm 兼容 |
| SHA256/HMAC | `sha2` + `hmac` | latest | 純 Rust 實現,用於 JWT HS256 和密碼 MD5 |
| MD5 | `md5` crate 或 `sha2` 生態 | latest | 兼容 PbootCMS 雙 MD5 密碼 |
| Base64 | `base64` crate | latest | JWT 編碼用 |
| UUID | `uuid` crate | latest | 生成 token 和唯一標識 |
| 時間 | `chrono` 或 `worker` 內置 | latest | 時間處理 |
| 日誌 | `worker` 的 `console_log!` | 內置 | Workers 環境日誌輸出 |

### 不使用的依賴及理由

| 依賴 | 不使用理由 |
|------|-----------|
| `sqlx` | D1 是 binding 資源,非網絡數據庫,sqlx 無法編譯到 wasm |
| `jsonwebtoken` | 依賴 ring/openssl C 庫,wasm 兼容性差,自行實現 HS256 更輕量 |
| `bcrypt` | 與 PbootCMS 原版雙 MD5 密碼不兼容 |
| `image` | 圖片處理會大幅增加 WASM 體積,水印交由靜態生成層處理 |
| `tokio` | Workers 有自己的異步運行時,不需要 tokio |
| `reqwest` | 用 `worker::Fetch` 替代,Workers 原生 HTTP 客戶端 |
| 任何模板引擎 | 後端純 API,不渲染頁面 |

### 管理後台 SPA

| 組件 | 選型 |
|------|------|
| 框架 | React 18 + TypeScript |
| UI 庫 | shadcn/ui + Tailwind CSS |
| 路由 | React Router v6 |
| HTTP 客戶端 | fetch + 自定義封裝 |
| 狀態管理 | Zustand 或 React Context |
| 構建工具 | Vite |
| 部署 | Cloudflare Pages |

### 靜態生成層

| 組件 | 選型 |
|------|------|
| 語言 | Go 或 Node.js（取決於已有技術棧） |
| 模板引擎 | Go html/template 或 Handlebars/Nunjucks |
| HTTP 客戶端 | 標準庫 fetch/http |
| 部署位置 | 香港自有伺服器 |
| 輸出 | 全靜態 HTML + 靜態資源 |
| 分發 | 同步至 CDN（可選 Cloudflare R2 + Pages） |

### 設計原則

1. **Handler 層薄**:只做參數提取和響應組裝,業務邏輯下沉到 Service
2. **Service 層厚**:所有 D1 查詢和業務規則在 Service 層,可復用
3. **Model 層純數據**:結構體只做字段映射,不含邏輯
4. **靜態 SQL**:SQL 語句以字符串常量定義,不用動態拼接
5. **零反射**:所有數據庫映射手動實現,不依賴反射

## 五、請求處理流程

```
請求進入
  │
  ├─ CORS 預檢 → 直接返回 204
  │
  ├─ /api/v1/auth/* → 認證處理
  │    ├─ login → 驗證密碼(雙MD5) → 簽發JWT → 返回token
  │    └─ 其他 → JWT驗證中間件
  │
  ├─ /api/v1/ssg/* → 靜態生成接口(API Key驗證)
  │
  ├─ /api/v1/* (公開接口) → 直接處理
  │    └─ 從KV讀取配置緩存 → 查詢D1 → 返回JSON
  │
  └─ /api/v1/admin/* (管理接口) → JWT鑑權中間件
       ├─ 驗證JWT簽名和過期時間
       ├─ 從KV檢查Token黑名單
       ├─ 權限校驗(超級管理員跳過)
       └─ 處理業務 → 寫D1 → 更新KV緩存 → 返回JSON
```

### 緩存策略

| 數據類型 | KV Key 格式 | TTL | 失效時機 |
|---------|-------------|-----|---------|
| 系統配置 | `config:all` | 永久 | 配置修改時刪除 |
| 欄目樹 | `sorts:tree` | 永久 | 欄目增刪改時刪除 |
| 站點信息 | `site:info` | 永久 | 站點信息修改時刪除 |
| 導航樹 | `nav:tree` | 3600s | 欄目修改時刪除 |
| 文章詳情 | `content:{id}` | 1800s | 文章修改時刪除 |
| Token黑名單 | `token:black:{jti}` | Token剩餘有效期 | 自動過期 |

### D1 查詢優化

1. **配置優先讀 KV**:系統配置、欄目樹等熱點數據優先從 KV 讀取,KV 未命中時回退 D1
2. **遞歸 CTE 查子孫欄目**:用 `WITH RECURSIVE` 一次查詢替代 N 次遞歸
3. **批量查詢消除 N+1**:文章列表分頁後,批量查詢擴展字段和欄目信息
4. **Count 優化**:統計總數時清除 ORDER BY,減少排序開銷
5. **索引覆蓋**:高頻查詢字段建立複合索引,避免回表

## 六、JWT 自行實現方案

JWT HS256 的核心是 HMAC-SHA256,用 `sha2` + `hmac` crate 即可實現:

```rust
// 簽發 Token
fn sign_jwt(payload: &JwtPayload, secret: &str) -> String {
    let header = base64url_encode(r#"{"alg":"HS256","typ":"JWT"}"#);
    let claims = base64url_encode(serde_json::to_string(payload).unwrap());
    let signing_input = format!("{}.{}", header, claims);
    let signature = hmac_sha256(signing_input.as_bytes(), secret.as_bytes());
    format!("{}.{}", signing_input, base64url_encode(&signature))
}

// 驗證 Token
fn verify_jwt(token: &str, secret: &str) -> Option<JwtPayload> {
    let parts: Vec<&str> = token.split('.').collect();
    if parts.len() != 3 { return None; }
    let signing_input = format!("{}.{}", parts[0], parts[1]);
    let expected_sig = hmac_sha256(signing_input.as_bytes(), secret.as_bytes());
    let actual_sig = base64url_decode(parts[2]);
    if !constant_time_eq(&expected_sig, &actual_sig) { return None; }
    serde_json::from_str(&base64url_decode_to_string(parts[1])).ok()
}
```

Token Payload:
```json
{
  "sub": "1",           // 管理員 ID
  "username": "admin",
  "exp": 1735689600,    // 過期時間(7天後)
  "iat": 1735084800,    // 簽發時間
  "jti": "uuid-v4"      // 唯一標識(用於黑名單)
}
```

## 七、WASM 體積控制策略

目標:編譯後 WASM 體積 ≤ 800KB。

### 控制手段

1. **Release 優化**:`Cargo.toml` 配置 `[profile.release]` 啟用 LTO 和優化
2. **最小依賴**:每個依賴都經過審查,不引入不必要的 transitive dependencies
3. **feature 裁剪**:只啟用需要的 feature flag
4. **wasm-opt**:編譯後用 `wasm-opt -Oz` 壓縮
5. **無模板引擎**:不引入任何模板渲染庫,大幅減少體積
6. **無圖片處理**:不引入 image crate
7. **自行實現 JWT**:不引入 jsonwebtoken 的依賴鏈

### Cargo.toml 優化配置

```toml
[profile.release]
opt-level = "z"      # 最小體積優化
lto = true            # 鏈接時優化
codegen-units = 1     # 單代碼生成單元,更好的優化
panic = "abort"       # panic 直接終止,減少 unwind 代碼
strip = true          # 移除調試符號
```

## 八、安全設計

### 認證安全

1. **密碼存儲**:雙 MD5（兼容 PbootCMS 原版）,使用常量時間比較
2. **登錄限頻**:基於 KV 的 IP 粒度限頻,5次失敗鎖定 15 分鐘
3. **JWT 過期**:默認 7 天,支持刷新機制
4. **Token 黑名單**:登出時將 JWT 的 jti 寫入 KV,過期後自動清理

### 輸入安全

1. **SQL 注入防護**:D1 binding API 天然參數化,所有用戶輸入通過 `.bind()` 傳遞
2. **XSS 防護**:用戶輸入的 HTML 內容在 API 層不做過濾（由靜態生成層的模板引擎負責轉義）
3. **參數校驗**:所有接口入參進行類型和長度校驗
4. **CORS**:僅允許配置的管理後台域名

### 訪問控制

1. **公開接口**:前台數據查詢接口無需鑑權
2. **管理接口**:所有後台管理接口需要 JWT
3. **超級管理員**:uid=1 跳過權限檢查
4. **角色權限**:基於 `ay_role_level` 的 URL 級權限校驗
5. **SSG 接口**:獨立的 API Key 驗證,與管理員 Token 體系隔離

## 九、部署架構

### 域名規劃

| 域名 | 指向 | 用途 |
|------|------|------|
| `api.example.com` | Worker | API 接口 |
| `admin.example.com` | Pages | 管理後台 SPA |
| `www.example.com` | CDN/靜態 | 前台靜態頁面 |
| `assets.example.com` | R2 公開訪問 | 靜態資源/圖片 |

### 部署流程

1. **Worker 部署**:`wrangler deploy` 一鍵部署,自動編譯 Rust → WASM
2. **D1 創建**:`wrangler d1 create` 創建數據庫,執行 `migrations/001_init.sql`
3. **KV 創建**:`wrangler kv namespace create` 創建命名空間
4. **R2 創建**:`wrangler r2 bucket create` 創建存儲桶
5. **Pages 部署**:管理後台 SPA `wrangler pages deploy` 部署
6. **DNS 配置**:將域名解析到對應的 Cloudflare 服務

### 環境隔離

- **開發環境**:`wrangler dev` 本地開發,使用本地 D1 和 KV 模擬
- **測試環境**:獨立的 Worker + D1 + KV,用於集成測試
- **生產環境**:正式的 Worker + D1 + KV + R2
